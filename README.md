# Midnight Appshop – NDI Reference Architecture

> "I went on an App shopping spree and saved thousands."

This repository contains the core reference architecture for **Midnight Appshop**, a decentralized, privacy-preserving application marketplace built on top of the Cardano ledger and the Midnight sidechain.

## 1. Executive Summary & Core Value
Midnight Appshop introduces a paradigm shift in software licensing: **Yield-Financed Utility with Zero-Knowledge Privacy**. It replaces the 2000s era monopoly model (Apple App Store / Google Play) characterized by high fees ("The 30% Tech Tax") and invasive user data harvesting.

*   **Capital Preservation:** Users retain 100% custody of their capital in a non-custodial wallet. The underlying staking yield algorithmically finances software access. Software utilization changes from an operational expense to a liquidity allocation strategy.
*   **Total Anonymity:** Utilizing Midnight's Zero-Knowledge Proofs (ZK-SNARKs), users prove eligibility (e.g., staking ≥ 680 ADA to a specific validation pool) without revealing public keys, transaction histories, or total asset net worth.

## 2. Protocol-Level Financial Yield-Splitting
Instead of traditional cash subscriptions, the native block rewards generated by the user's non-custodial delegation are split and distributed algorithmically at the protocol level into four separate buckets:

1.  **The Developer (e.g., Sisoft):** Receives an agreed fixed FIAT-equivalent share from the staking reward pool (e.g., 10 NOK per 30 days per active user session).
2.  **The SaaS Provider (The Platform Operator):** Takes a set percentage fee for running the storefront, providing the legal MiCA/CASP framework, and managing FIAT off-ramping.
3.  **The IaaS Provider (Seize it AS):** Obtains the native SPO block rewards and the exact allocation of **DUST** tokens required to power, host, and validate the target dApp logic.
4.  **The End User (The Customer):** Retains the remaining native ADA and NIGHT rewards in full custody, while delegating the necessary DUST to the specific isolated Docker container running the web application.

## 3. Decoupled Repository Architecture

To ensure enterprise-grade security, scalability, and simplified deployment, the system architecture separates responsibilities across physical nodes. 

┌────────────────────────────────────────────────────────┐│               FINANCIAL & STOREFRONT LAYER             ││        [dell_server] Running Apache & MariaDB          │└───────────────────────────┬────────────────────────────┘│ Secure HTTPS / API Localnet▼┌────────────────────────────────────────────────────────┐│             DECENTRALIZED CRYPTO/VALIDATION            ││       [shaun3_node] Isolated Ubuntu Network Node       │└────────────────────────────────────────────────────────┘
### Directory Structure & Deployment Breakdown:

*   **`database/`**  
    Contains `schema.sql` to initialize the relational metadata store. Runs exclusively on the financial host (**Dell**). Handles app listings, active anonymous session states, and ledger billing records.
*   **`dell_server/`**  
    The core storefront application layer. Handles user routing, updates active MariaDB session tables, and contains the `business20_api.php` module for automated ledger alignment with enterprise financial accounting packages (**Business 2.0 - Module 5 / Kryptoforvaltning**). 
    *   *Deployment Note:* A SaaS App Store operator **only** requires the code in `database/` and `dell_server/`.
*   **`shaun3_node/`**  
    The isolated cryptographic infrastructure layer. Runs on a hardened Ubuntu target host (**shaun3**). Manages connections to the Midnight sidechain/Preprod testnet and executes the Compact ZK smart contract circuits. 
    *   *Deployment Note:* This component runs strictly in the background as an infrastructure utility managed by the IaaS vendor (Seize it AS). Storefront operators do not need to host or interact with this codebase directly.

## 4. Local Evaluation & Testing
To evaluate the protocol flow without the overhead of heavy indexed ledgers on lightweight hardware, this package provides an end-to-end network simulation framework.

1.  **Crypto Validator Layer (shaun3):** Gaining SSH access to the infrastructure layer, navigate to `shaun3_node/`, run `npm install`, and launch the Express microservice via `node server.js` to begin listening on port 3000.
2.  **Application Layer (Dell):** Initialize the schema on your MariaDB instance. Set up a secure database administrative credential (`Appshop_Admin`). Configure `verify.php` to direct outbound data transport requests over the local area network to the active node endpoint.
3.  **Enterprise Sync:** Query `business20_api.php` utilizing your secure system token to fetch cryptographically logged, split financial bills prepared for instant ledger ingestion.
